# ТЗ для Codex — Fish Olha Desktop Catalog Manager v1

## 0. Контекст проекта

Есть действующий сайт магазина рыбы и морепродуктов **fisholha.ru** на **WordPress + WooCommerce**.

Нужно создать **desktop-приложение для Windows** для владельца магазина, которое будет использоваться как основной локальный инструмент управления каталогом и фотографиями товаров.

Приложение:
- работает локально на ноутбуке владельца;
- хранит локальную рабочую базу данных;
- позволяет загружать текущий каталог из WooCommerce;
- позволяет редактировать категории, товары и фото;
- поддерживает массовое редактирование;
- работает по модели **draft → publish**;
- выгружает изменения обратно в WooCommerce;
- готово к будущему расширению в другие каналы.

# 1. Главная цель MVP

Создать **простое, понятное, современное desktop-приложение**, которым может пользоваться владелец магазина с низким уровнем IT-подготовки.

Фокус MVP:
1. импорт каталога из WooCommerce
2. локальное хранение каталога

- Python 3.12+
- PySide6

### Локальная БД
- SQLite

### Работа с изображениями
- Pillow
- локальное файловое хранилище

### Интеграция с WooCommerce
- WooCommerce REST API
- requests или httpx

### Конфигурация
- pydantic / pydantic-settings
или эквивалентный аккуратный подход

## 4.2 Почему PySide6
PySide6 подходит, потому что:
- позволяет сделать современный desktop UI;
- поддерживает таблицы, формы, дерево категорий, drag-and-drop;
- подходит для single-window приложения;
- удобен для дальнейшего масштабирования.

## 4.3 Почему SQLite
Для MVP SQLite подходит, потому что:
- приложение локальное;
- один основной пользователь;
- простая установка;
- нет необходимости поднимать отдельный сервер БД.

# 5. Основная концепция работы

## 5.1 Главный принцип
WooCommerce сейчас является публичной витриной сайта, но не должен быть единственным рабочим редактором данных.

Новая система становится:
- локальным рабочим кабинетом;
- редактором каталога;
- хранилищем черновиков;
- точкой публикации на сайт.

## 5.2 Data flow
### На старте
WooCommerce → импорт в локальную БД приложения

### После этого
Редактирование происходит локально в приложении

### Затем
Локальные изменения публикуются в WooCommerce

## 5.3 Важный принцип
Нельзя автоматически отправлять каждое изменение сразу на сайт.

Нужен workflow:
- локальные изменения
- статус "изменено"
- выбор изменений к публикации
- публикация
- лог результата

# 6. Авторизация

## 6.1 MVP-решение
Сделать **локальную авторизацию** внутри приложения:
- логин
- пароль

## 6.2 Почему не SSO с WooCommerce
Для desktop MVP SSO с WooCommerce не нужен, потому что:
- лишняя сложность;
- лишняя зависимость от WordPress auth;
- усложняет UX владельцу;
- не дает критической пользы.

## 6.3 Требования к локальной авторизации
- экран входа при запуске приложения
- безопасное хранение хеша пароля
- возможность смены пароля
- минимальный локальный user management foundation

# 26. Критерии успеха MVP

MVP считается успешным, если владелец магазина может:
1. открыть desktop-приложение
2. войти по логину/паролю
3. загрузить текущий каталог из WooCommerce
4. увидеть категории и товары
5. отредактировать товары и категории
6. массово отредактировать выбранные товары
7. прикрепить/изменить фото
8. увидеть, что изменено локально
9. выгрузить изменения в WooCommerce
10. понять, что произошло, через журнал операций
